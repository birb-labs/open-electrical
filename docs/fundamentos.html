Fundamentos

Ambiente, estrutura, stack e a lógica geral de funcionamento do plugin.

Requisitos

O open-electrical é um módulo nativo compilado (não é LISP), então exige o BricsCAD Pro ou superior (a edição Lite carrega apenas LISP). Foi desenvolvido e testado para BricsCAD V26 em Linux (também compila para Windows).

Build systemCMake + NinjaConfiguração e compilação
Por que wxWidgets ESTÁTICO no Linux? O BricsCAD já carrega o seu próprio wxWidgets (3.1). Linkar o módulo contra um wxWidgets compartilhado do sistema coloca dois runtimes wx no mesmo processo e derruba o host ao carregar. Por isso o CMake linka um wxWidgets privado e estático (prefixo wxstatic-oel-*).

Compilar e instalar

Há um script que configura, compila e instala em um passo:

scripts/setup.sh 
bash scripts/setup.sh
# → build/ (Ninja) → open-electrical.lrx
# → instala em ~/.local/share/open-electrical/

Rodar os testes de unidade (núcleo sem BRX):

cmake --build build --target check   # executa oe_tests

Carregar no BricsCAD:

APPLOAD  →  ~/.local/share/open-electrical/open-electrical.lrx
; ou na linha de comando:
(arxload "/home/<user>/.local/share/open-electrical/open-electrical.lrx")
Cache de carga (.lrx). O BricsCAD faz cache do módulo em nível de processo. Depois de recompilar, feche e reabra o BricsCAD — um simples re-APPLOAD pode continuar executando o binário antigo.

│   ├── lang/               # pt-BR.json, en.json, es.json (i18n)
│   ├── symbols/            # blocos de símbolos
│   └── ui/dcl/             # diálogos DCL
├── tests/                  # test_framework.h + tests.cpp (núcleo sem BRX)
└── src/
    ├── main.cpp            # entrypoint ARX (IMPLEMENT_ARX_ENTRYPOINT)
    ├── Platform.h          # inclui os headers BRX/ARX
    ├── commands/           # comandos EL-* (a "camada de UI de linha de comando")
    ├── models/             # dados puros: Room, Circuit, Conduit, Outlet, ProjectData…
    ├── services/           # regras: Router, CircuitDistributor, LightingCalculator…
    ├── ui/                 # paletas wxWidgets, diálogos, i18n
    └── utilities/          # helpers geométricos: ElementOutline, PolyMath, Geometry…

Camadas (fluxo de uma dependência)

commands → services → models / utilities. Os arquivos em commands/ conversam com o BricsCAD (pegar pontos, desenhar entidades, imprimir na linha de comando) e delegam a lógica para services/, que operam sobre estruturas puras de models/. Os utilities/ (ex.: geometria) não dependem do BRX e por isso são cobertos por testes de unidade.

Lógica geral de funcionamento

O plugin registra uma tabela de comandos EL-* e mantém um modelo de projeto (ProjectData) que espelha, em memória, tudo o que foi inserido no desenho: cômodos, componentes, circuitos e eletrodutos.

1. Registro de comandos

Todos os comandos ficam numa tabela única e são registrados automaticamente.

src/commands/Commands.cpp 
{ _T("EL-CIRCUIT-AUTO"),_T("EL-CIRCUIT-AUTO"),&distributeCircuitsAuto, ACRX_CMD_MODAL },
{ _T("EL-CONDUIT-AUTO"),_T("EL-CONDUIT-AUTO"),&routeConduitAuto,       ACRX_CMD_MODAL },
{ _T("EL-WIRE"),        _T("EL-WIRE"),        &routeWire,              ACRX_CMD_MODAL },
{ _T("EL-WIRE-AUTO"),   _T("EL-WIRE-AUTO"),   &routeWireAuto,          ACRX_CMD_MODAL },
{ _T("EL-WIRE-FLIP"),   _T("EL-WIRE-FLIP"),   &flipWiring,  ACRX_CMD_MODAL | ACRX_CMD_USEPICKSET },
// ... (adicionar um comando = 1 linha aqui + 1 declaração em Commands.h)

2. O modelo ProjectData

É o "documento" do plugin. Cada componente/eletroduto guarda o handle da sua entidade no desenho, então o modelo e o desenho ficam sincronizados por handle.

src/models/ProjectData.h (resumo) 
struct ProjectData {
    std::vector<Room>                              rooms;
    std::vector<std::unique_ptr<ElectricalElement>> elements; // tomadas, luzes, interruptores, quadros
    std::vector<Circuit>                           circuits;   // ramais
    std::vector<Conduit>                           conduits;   // eletrodutos
    ProjectSettings settings;
    // + alloc*Id() para ids incrementais
};

3. Persistência

O ProjectContext (singleton) serializa o ProjectData para dentro do próprio DWG (via PropertyBag → dicionário/named object). Cada save() grava; ao abrir o desenho, o estado é reconstruído. Por isso a identificação de circuitos não usa camadas ou atributos de bloco: ela vive no modelo, ligada aos componentes por circuitId.

4. Um comando típico, do começo ao fim

  1. O usuário digita EL-…; o BricsCAD chama a função registrada.
  2. A função pega ProjectContext::instance().project().
  3. Abre um UndoGroup (para o comando ser um único "desfazer").
  4. Chama um service (ex.: Router) que altera o modelo.
  5. Desenha as entidades resultantes (helpers em CommandUtil.cpp).
  6. ProjectContext::instance().save() e imprime um resumo.
Pronto para o mergulho profundo? Vá para EL-WIRE / EL-WIRE-AUTO — o comando que calcula os condutores por circuito e desenha os balões dentro dos eletrodutos.
